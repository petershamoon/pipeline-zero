PipelineZero - ContractFlow Build Plan v2.0

Last Updated: March 6, 2026

Purpose: Complete build specification for a secure, demo-ready ContractFlow platform with explicit production-safe and demo-safe modes.

Document intent: This version replaces the earlier plan and fixes architecture, security, and deployment risks while preserving the original demo narrative.

1. Executive Summary

  • Keep a modular monolith architecture: FastAPI backend + React frontend + PostgreSQL.
  • Move scheduled business jobs out of API process into Azure Container Apps Jobs (or Azure Functions Timer).
  • Use cookie-based auth session model for browser clients (HttpOnly, Secure, SameSite, CSRF protection).
  • Use distributed rate limiting and token/session controls backed by Redis.
  • Use GitHub OIDC federation to Azure; remove long-lived cloud credentials from GitHub secrets.
  • Pin all GitHub Actions by full commit SHA, not mutable tags like @master.
  • Split workflows into Production Safe and Demo Vulnerability modes to avoid accidental insecure releases.

4. Architecture Decisions

4.1 Core pattern

  • Use a modular monolith with strong domain boundaries: auth, contracts, versions, approvals, admin, audit.
  • Avoid premature microservices. Keep one deployable backend service plus independent scheduled jobs.
  • Extract heavy background tasks into jobs, not API startup loops.

4.2 Request flow

  • Frontend sends request with secure session cookie.
  • API middleware validates session/JWT and loads user context.
  • Endpoint executes role + object-level authorization checks.
  • Service layer performs domain logic.
  • Repository/ORM handles persistence and transaction scope.
  • Audit event is appended for all state-changing actions.

4.3 Scheduling model

  • Run expiration and reminder tasks with Azure Container Apps Jobs on cron schedule.
  • Use idempotency key and advisory locks to prevent duplicate updates.
  • Do not run perpetual asyncio.sleep() loops in API containers.

5. Security Architecture

5.1 Authentication and sessions

  • Preferred: Entra ID with OIDC authorization code flow + PKCE for SPA.
  • Fallback: local auth with Argon2id password hashes and short-lived session cookies.
  • Store refresh/session tokens server-side as hashes and rotate on every use.
  • Never persist bearer tokens in localStorage or sessionStorage.

5.2 Authorization

  • Enforce both role-based and object-level checks (BOLA prevention).
  • Centralize authorization logic in reusable policy functions and test them directly.
  • UI may hide unauthorized controls, but API remains final enforcement boundary.

5.3 Upload hardening

  • Validate MIME type using python-magic and allowlist only approved types.
  • Perform max-size checks before persistence and use streamed uploads where practical.
  • Store SHA-256 checksum and optionally route uploaded files through malware scanning.
  • Issue short-lived SAS URLs scoped to read-only operations.

5.4 Application hardening

  • Security headers: HSTS, X-Content-Type-Options, Referrer-Policy, frame restrictions.
  • CORS set to explicit trusted origins only.
  • Global exception handler returns opaque errors; stack traces only in logs.
  • Rate limits per route and per identity/IP with distributed backend.

6. Data Model Updates

6.1 Required schema controls

  • Use UUID primary keys, timezone-aware timestamps, and soft delete where needed.
  • Add check constraints for date ranges and non-negative currency values.
  • Add optimistic locking/version field for concurrent contract edits.
  • Add unique index on normalized email (case-insensitive).
  • Rename audit field from metadata to metadata_json in ORM model to avoid SQLAlchemy conflicts.

6.2 Audit model requirements

  • Append-only audit table with immutable rows.
  • Capture actor, action, resource, IP, user-agent, correlation id, and metadata payload.
  • Write audit event in same transaction when business mutation occurs.

7. Backend Build Specification

7.1 Config and startup

  • Use Pydantic v2 settings with explicit parsing for list values (for origins, hosts).
  • Support local .env only in development mode.
  • Expose /health/live and /health/ready endpoints.

7.2 API design and standards

  • REST endpoints under /api/v1.
  • Consistent error envelope and error codes.
  • Pagination: cursor or limit/offset with total count for admin/reporting views.
  • All state changes produce audit events.

7.3 Docker hardening

  • Run as non-root user.
  • Include only required OS packages and clean apt cache.
  • Use JSON-form CMD syntax with double quotes.
  • Ensure healthcheck command exists in image (install curl or use python probe).

7.4 Testing requirements

  • Unit tests for services and authorization policies.
  • Integration tests for auth, contract lifecycle, approvals, and upload flow.
  • Security tests for unauthorized access and policy bypass attempts.
  • Target baseline coverage of 70 percent on new code.

8. Frontend Build Specification

8.1 Session handling

  • Use secure cookie auth; frontend should not read refresh token directly.
  • Keep only non-sensitive user profile and UI preferences in persisted state.
  • Use React Query for server state and cache invalidation after mutations.

8.2 UX and security rules

  • Render role-appropriate views and actions.
  • Gracefully handle 401/403 with redirect and user feedback.
  • Client-side file checks are UX hints only; server is authority.

8.3 Nginx container

  • Add explicit /health location returning 200 for probes.
  • Serve immutable static assets with long cache headers.
  • Retain SPA fallback routing to index.html.

9. CI/CD and Security Pipeline

9.1 Pipeline stages

  • Stage 1: format and lint.
  • Stage 2: unit and integration tests.
  • Stage 3: SAST/SCA/secret scanning (Semgrep, CodeQL, Trivy, Gitleaks).
  • Stage 4: build images, generate SBOM, sign images.
  • Stage 5: deploy to staging.
  • Stage 6: DAST against staging.
  • Stage 7: manual approval gate for production promotion.

9.2 GitHub Actions requirements

  • Pin all third-party actions by commit SHA.
  • Use least-privilege job permissions.
  • Use OIDC login to Azure; remove static AZURE_CREDENTIALS secret.
  • Use workflow concurrency controls for deployment workflows.

9.3 Semgrep rule strategy

  • Refine custom rules to avoid brittle pattern matching and false positives.
  • Add regression tests for each custom rule with positive/negative fixtures.
  • Treat auth bypass, raw SQL, hardcoded secret, and insecure CORS as blocking findings.

10. Infrastructure Plan (Terraform)

10.1 Provisioned resources

  • Resource Group, VNet/subnets, private DNS where required.
  • Container Apps Environment for backend, frontend, and jobs.
  • PostgreSQL Flexible Server (private access), Redis, Blob Storage, Key Vault.
  • ACR with admin disabled and pull via managed identity.
  • Log Analytics + Azure Monitor alerts.

10.2 Environment strategy

  • Separate staging and production via dedicated variable sets and state isolation.
  • Use remote Terraform backend with state locking.
  • Tag all resources by environment, owner, data-classification, and cost-center.

10.3 Reliability and recovery

  • Configure automated Postgres backups and retention policy.
  • Define RPO/RTO targets and restore runbook.
  • Enable alerting for failed jobs, high error rates, and auth anomaly spikes.

11. Demo Mode vs Production Mode

Critical control: seeded vulnerabilities are allowed only in Demo Mode branches/workspaces and must never be merged into protected production branches.
  • Create dedicated branch family: demo/vuln-*.
  • Add branch protection so demo branches cannot deploy to production environments.
  • Add repository labels and PR templates that mark intentional vulnerabilities.
  • Use separate staging namespace for demo scans if possible.

12. Build Order (Revised)

  1. Repository scaffold, base Docker Compose, health endpoints.
  2. Models, migrations, indexes, and constraints.
  3. Auth/session implementation and authorization policy library.
  4. Contract CRUD with object-level access checks.
  5. Versioned file upload/download with MIME validation and checksums.
  6. Approval chain and template engine.
  7. Audit pipeline and admin reporting APIs.
  8. React auth and shell pages, then contract workflows.
  9. Background jobs and idempotency protections.
  10. CI/CD with tests + security gates + staging deploy.
  11. DAST and observability dashboards.
  12. Demo mode vulnerability seeding in isolated branch only.

13. Non-Negotiable Acceptance Criteria

  • No plaintext secrets in repository or CI logs.
  • No auth tokens persisted in browser storage.
  • No in-app cron loops for multi-replica business tasks.
  • All protected endpoints validated by role and object-level checks.
  • CI pipeline blocks merge on critical security findings.
  • Staging deployment verifies health, migration, and DAST before release approval.

  • Operational runbook for incident response and key rotation.
